    GNSS Testbed Simulator¶

    A GPU-accelerated, Python-based GNSS signal simulation engine.
    High-fidelity. Multi-constellation. Multi-band. Multi-element.

    ⚡

    GPU-Accelerated¶

    Fused CUDA kernels map the entire multi-band antenna array across thousands of parallel execution threads. Frame generation runs in O(1) time — executing with uncompromising, constant performance independent of total simulation duration.

    🛰️

    Multi-Constellation & Multi-Band¶

    Simultaneously generate GPS L1 C/A, GPS L2C, and Galileo E1 (CBOC) signals from all visible satellites. GPS L5, BeiDou, and GLONASS support planned.

    📡

    Antenna Array Simulation¶

    Arbitrary M-element antenna arrays with per-element lever arms, orientations, and radiation patterns. Built for CRPA and adaptive null-steering research.

    🎯

    Full 6-DoF Dynamics¶

    Polynomial trajectory model (position + velocity + acceleration + jerk) for both translational and rotational motion. Receiver, jammers, and spoofers all support full kinematics.

    🔴

    Rich Interference Library¶

    Built-in jammer types: CW, Triangle FM, BPSK, Chirp, and Broadband Noise. Each with independent trajectory, power, frequency, and antenna pattern.

    🌍

    Physics-Based Channel¶

    Saastamoinen tropospheric model + Klobuchar ionospheric model. Polyphase fractional-delay resampling with sub-sample accuracy. Dual delay paths (code vs. carrier) for realistic iono dispersion.

    🐍

    Python-Native & Extensible¶

    Built entirely in Python with CuPy/CUDA acceleration. Adding a new constellation, signal type, or jammer takes hours — not months. Configuration-driven via clean YAML files.

    📊

    Production Monitoring¶

    Real-time Prometheus metrics for frame generation time, throughput, and GPU utilization. Grafana dashboards for live performance visualization during simulation runs.

    📈

    Future-Proof Scalability¶

    Built for the extreme edge of Electronic Warfare. Exhibiting powerful sub-linear geometric scaling, the engine naturally spans from single-GPU workstations to distributed DGX supercomputers for massive, real-time 16-element arrays.

    Quick Start: 60 Seconds to Signals¶

    Generate complex, multi-band GNSS environments with two simple commands.

    # 1. Start the GPU generation engine (Server)
python -m gnss_sim.framework.server --params config/simulator_params.yaml

# 2. In a separate terminal, consume the signals and write to disk (Client)
python experiments/rcv_pocketsdr_phase5.py --scenario config/scenario_config.yaml

    Fully Extensible

    Don't want to use the default Python client? You can write a custom C++ or Python script to consume the ZeroMQ output stream and pipe the raw IQ data directly into your own Software-Defined Radio (SDR).

    Architecture at a Glance¶

    The simulator uses a client-server architecture over ZeroMQ, fully decoupling signal generation from signal consumption.

    graph LR
    subgraph Client["🖥️ Client (Tester)"]
        A[Configure Scenario] --> B[Request Frames]
        B --> C[Receive IQ Data]
        C --> D[Add Noise / Quantize]
        D --> E[Write to File]
        E --> F[Feed Your Receiver]
    end

    subgraph Server["⚙️ Server (GPU Engine)"]
        G[Load Config & Ephemerides] --> H[Compute Geometry]
        H --> I[Generate Signals on GPU]
        I --> J[Transfer to Host]
    end

    B -- "ZMQ REQ/REP" --> G
    J -- "Multi-part Binary" --> C

    The server focuses exclusively on signal generation: geometry computation, GPU kernel dispatch, and high-precision baseband rendering. It outputs full-precision complex64 IQ samples — no quantization, no noise.

    The client adds thermal noise, applies AGC, quantizes, and writes output files that can be consumed by any software or hardware GNSS receiver.

    Traditional GNSS simulators require specialized real-time hardware that costs over $1,000,000 for multi-channel, multi-band, phase-coherent simulation. This architecture achieves equivalent fidelity using commodity GPU servers and a Python-based software engine — at a fraction of the cost.

    The key insight: for 95% of verification tasks, real-time simulation is unnecessary. By accepting a slowdown factor (e.g., ×10–×20), we trade "cheap" compute time for "expensive" RF capability. A scenario that runs overnight delivers $1M worth of simulation fidelity on $65K worth of hardware.

    O(1) Frame Processing¶

    The simulator processes each frame in constant time, regardless of whether the simulation runs for 1 second or 1 hour.

    What Scales What Does NOT Scale
    Total wall-clock time (linear with duration) Per-frame GPU processing time
    Total output data volume Memory usage per frame
    Kernel launch overhead

    This means doubling the simulation duration exactly doubles the wall-clock time — no hidden overhead, no memory growth, no degradation. The performance benchmarks section provides measured per-frame timing data across various scenarios and GPU tiers.

    PocketSDR-Validated ✅¶

    The simulator's signal quality is validated end-to-end against PocketSDR, an open-source multi-band GNSS software receiver. This means:

    1. Simulator generates multi-band IQ data (complex64, full precision)
    2. Quantized output is fed directly to PocketSDR
    3. PocketSDR successfully acquires and tracks all simulated satellites
    4. CN0, Doppler, and code phase match expected values

    This "PocketSDR-approved" stamp ensures that the simulator's output is not just mathematically correct — it is receivable and decodable by real GNSS processing software.

    Your receiver, your choice

    PocketSDR is used internally for validation. You are free to use the simulator's output with any receiver — software or hardware. PocketSDR installation is not required to use the simulator.
