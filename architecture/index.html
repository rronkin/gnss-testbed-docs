    Architecture¶

    Overview¶

    The GNSS Testbed Simulator is built around a client-server model that separates signal generation (compute-intensive, GPU-bound) from signal consumption (flexible, user-defined). The two processes communicate over ZeroMQ using a synchronous request-reply protocol.

    Client-Server Protocol¶

    The protocol follows a simple ping-pong pattern:

    Command Reference¶

    Command Direction Description
    INIT Client → Server Start async initialization with scenario config path
    STATUS Client → Server Poll initialization progress
    GENERATE Client → Server Request N frames of signal data
    STOP Client → Server Graceful shutdown

    Initialization Sequence¶

    sequenceDiagram
    participant C as Client
    participant S as Server

    C->>S: INIT (scenario path, work_dir)
    S-->>C: OK — "Initialization started"

    loop Poll until READY
        C->>S: STATUS
        S-->>C: LOADING_CONFIG / PREPARING_EPHEMERIS / CALCULATING_VISIBILITY / INITIALIZING_GENERATOR / COMPILING_KERNELS
    end

    S-->>C: READY (fs, M, start_time_gps_sec)

    The server uses a background thread for initialization, allowing the client to poll progress without blocking. The status phases provide visibility into what the server is doing:

    Phase Description
    LOADING_CONFIG Parsing YAML configuration files
    PREPARING_EPHEMERIS Loading RINEX navigation data
    DOWNLOADING_RINEX Auto-downloading RINEX if not provided locally
    CALCULATING_VISIBILITY Determining which satellites are above the elevation mask
    INITIALIZING_GENERATOR Building the signal generation pipeline
    COMPILING_KERNELS JIT-compiling CUDA kernels and running the first warm-up frame
    READY Server is ready to accept GENERATE requests

    Frame Generation Loop¶

    sequenceDiagram
    participant C as Client
    participant S as Server

    loop For each batch
        C->>S: GENERATE (n_frames=10)
        Note over S: GPU processes all frames
        S-->>C: JSON metadata + binary IQ buffers
        Note over C: Add noise, quantize, write
    end

    C->>S: STOP
    S-->>C: OK

    The server responds with a multi-part ZeroMQ message: JSON metadata followed by binary buffers for each [band × element × frame] combination.

    Frame Processing Pipeline¶

    Each frame covers a configurable time window (default: 100 ms). Within a frame, the server executes these stages:

    1. Receiver Dynamics¶

    Evaluate the receiver's 6-DoF polynomial trajectory at the required time points. This produces position, velocity, and attitude (as a Direction Cosine Matrix) at each coarse interpolation step (default: every 1 ms).

    2. Geometry Computation¶

    For every (satellite × antenna element) pair:

    • Compute propagation delay using iterative and quadratic solvers
    • Apply satellite clock corrections (bias + drift)
    • Compute tropospheric delay (Saastamoinen zenith delay + Niell mapping function)
    • Compute ionospheric delay (Klobuchar model, frequency-scaled per band)
    • Derive Doppler via numerical gradient of the corrected delay profile

    3. GPU Signal Generation¶

    For each active band, the fused CUDA kernel processes all satellites simultaneously:

    • Baseband generation: PRN code lookup, navigation bit modulation, subcarrier generation (CBOC for Galileo E1)
    • Polyphase resampling: Fractional-delay filtering using a pre-computed Kaiser-windowed filter bank
    • Carrier rotation: Phase-coherent carrier application using the phase-delay path
    • Accumulation: All satellites sum into a shared output buffer per antenna element

    4. Jammer Injection¶

    If interference emitters are configured, the jammer fleet kernel runs in the same GPU pass:

    • Per-jammer geometry computed on-the-fly (position → delay → Doppler)
    • Signal generation is type-specific (CW tone, FM sweep, BPSK spreading, etc.)
    • Output accumulates into the same master buffer alongside satellite signals

    5. Transfer & Response¶

    The GPU output buffers (one per band per antenna element) are transferred to host memory via pinned (page-locked) buffers and sent to the client over ZeroMQ.

    Multi-Band Architecture¶

    The simulator supports simultaneous multi-band generation. Each constellation-band combination maps to an RF band:

    Constellation-Band RF Band Carrier Frequency Default Sample Rate
    GPS L1 C/A L1 1575.42 MHz 6.138 MHz
    Galileo E1 (CBOC) L1 1575.42 MHz 6.138 MHz
    GPS L2C L2 1227.60 MHz 6.138 MHz

    Bands sharing the same RF band (e.g., GPS L1 and Galileo E1 both on L1) are summed into a common output buffer. Each RF band has its own sample rate and can be configured independently.

    Antenna Array Processing¶

    The simulator generates independent IQ streams for each antenna element. Each element has:

    • A lever arm (offset from the receiver's center of mass in body frame)
    • An orientation (Euler angles for boresight direction)
    • An optional radiation pattern (CST-compatible or isotropic)

    At each time step, the geometry engine computes element-specific delays by combining the receiver body dynamics with each element's lever arm offset, rotated through the current attitude DCM. This produces phase-coherent signals across all elements — exactly what CRPA and adaptive beamforming algorithms require.

    Monitoring¶

    The server exposes real-time Prometheus metrics:

    Metric Type Description
    frames_generated_total Counter Total frames generated since server start
    processing_time_seconds Summary Per-frame GPU processing time distribution

    These metrics can be visualized in Grafana dashboards for real-time monitoring of simulation performance, GPU utilization, and throughput during long-running scenarios.
