    Output Formats¶

    This page describes the data produced by the simulation server and the files generated by the client.

    Server Outputs¶

    The server generates full-precision, unquantized complex baseband signals and sends them to the client over ZeroMQ.

    Signal Buffers¶

    Output Shape Type Description
    IQ Signal [N_bands × M × N_samples] complex64 Baseband signal per RF band and antenna element
    Frame Metadata — JSON Frame index, time offset

    Where:

    • N_bands = number of active RF bands (e.g., 2 for L1+L2)
    • M = number of antenna elements
    • N_samples = frame_period_s × sample_rate_hz (samples per frame per band)

    No noise, no quantization

    The server output is a mathematically ideal signal (satellites + jammers) at full floating-point precision. Thermal noise, AGC, and quantization are applied on the client side, giving maximum flexibility for different test scenarios.

    Multi-Part ZeroMQ Message¶

    Each GENERATE response consists of:

    1. Part 0: JSON metadata 

      {
  "status": "OK",
  "n_frames": 10,
  "frames": [{"frame_idx": 0, "time_offset_s": 0.0}, ...],
  "shape": [1, 10, 613800],
  "bands": ["L1"]
}

    2. Parts 1..N: Binary buffers (complex64 arrays) in order: band[0]/element[0]/frame[0], band[0]/element[0]/frame[1], ..., band[N]/element[M]/frame[F]

    Client Outputs¶

    The reference client (rcv_pocketsdr_phase5.py) processes the server's raw IQ data and produces files ready for any GNSS receiver — software or hardware.

    Processing Pipeline¶

    graph LR
    A["Server IQ<br>(complex64)"] --> B["Add Thermal Noise<br>(-174 dBm/Hz)"]
    B --> C["AGC<br>(EWMA variance tracking)"]
    C --> D["Resample<br>(to PocketSDR rate)"]
    D --> E["Quantize<br>(2-3 bit)"]
    E --> F["Write .bin file"]
    F --> G["Generate .tag file"]

    Output Files¶

    For each antenna element m:

    File Format Description
    ant{m}.bin Binary Quantized IQ samples in PocketSDR format
    ant{m}.bin.tag Text PocketSDR metadata tag file

    Tag File Contents¶

    The .tag file tells PocketSDR how to interpret the binary data:

    [Channel 1]
path = ant0.bin
format = INT8X2
sampling_freq = 6138000.0
intermediate_freq = 0.0
f_lo = 1575420000.0
IQ = 2
bits = 8

    For dual-band scenarios, the format changes to RAW8 with interleaved L1/L2 samples:

    [Channel 1]
path = ant0.bin
format = RAW8
sampling_freq = 6138000.0
intermediate_freq = 0.0
f_lo = 1575420000.0,1227600000.0
IQ = 2,2
bits = 2,2

    Verification with PocketSDR (Optional)¶

    After writing the output files, the client can optionally run PocketSDR (an open-source GNSS receiver) for automated signal verification:

    Output Description
    rcv_ant{m}.log Tracking log with per-satellite CN0, Doppler, and code phase

    The tracking log is parsed to produce a consistency check across antenna elements:

    --- Multi-Antenna Consistency Check ---
Antenna  PRN    CN0    Doppler
      0    1  42.3    -1523.4
      0    3  39.8     2104.7
      1    1  42.1    -1523.5
      1    3  39.7     2104.6

PRN 01: CN0 spread = 0.2 dB-Hz, Doppler spread = 0.1 Hz
PRN 03: CN0 spread = 0.1 dB-Hz, Doppler spread = 0.1 Hz

    Ideal result

    For co-located antenna elements (zero lever arms), CN0 and Doppler should be nearly identical across all elements. Small spreads (< 0.5 dB CN0, < 1 Hz Doppler) confirm correct multi-element signal generation.

    Telemetry Data¶

    When pipeline.enable_telemetry is true, the server logs per-frame, per-satellite telemetry:

    Field Type Unit Description
    PRN string — Satellite vehicle ID (e.g., G01, E11)
    Band string — Constellation-band (e.g., GPS_L1)
    Time_s float s Simulation time at frame end
    Delay_s float s Propagation delay to element 0
    Doppler_Hz float Hz Carrier Doppler frequency
    Phase_cycles float cycles Carrier phase (accumulated)

    Prometheus Metrics¶

    The server exposes real-time metrics on port 8000:

    Metric Type Description
    frames_generated_total Counter Cumulative frame count
    processing_time_seconds Summary GPU processing time per frame (mean, quantiles)

    The client exposes metrics on port 8002:

    Metric Type Description
    requests_sent_total Counter Number of GENERATE requests
    network_latency_seconds Summary Round-trip time per request
