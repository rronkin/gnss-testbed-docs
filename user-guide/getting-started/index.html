    Getting Started¶

    Prerequisites¶

    • Python 3.11+ with a CUDA-capable GPU (for server-side signal generation)
    • Conda (Anaconda or Miniconda) for environment management
    • PocketSDR (optional, included in tools/) — for signal verification only

    Server Requirements¶

    The simulator server requires a CUDA-capable GPU. Performance scales primarily with GPU compute and memory bandwidth.

    Hardware¶

    Component Minimum Recommended
    GPU NVIDIA T4 / RTX 3060 (8 GB VRAM) NVIDIA A100 / RTX 4090 (24+ GB VRAM)
    CPU 4 cores 8+ cores
    RAM 16 GB 32+ GB
    Disk 10 GB free 50+ GB SSD (for long simulations)
    Network Localhost only 1 Gbps (if client is remote)

    Scaling factors

    VRAM usage scales with: number of antenna elements × number of active bands × sample rate × frame period. A 4-element, dual-band scenario at 24 MHz uses ~4× more VRAM than a single-element, single-band scenario at 6 MHz. The Performance Benchmarks will provide measured numbers as they are collected.

    Software¶

    Component Version
    OS Ubuntu 22.04+ (recommended) / Windows 10+
    CUDA Toolkit 11.8+
    Python 3.11+
    CuPy 13.0+ (matching CUDA version)
    ZeroMQ 4.3+ (via pyzmq)

    Environment Setup¶

    First-time setup: 

    bash scripts/setup_env.sh

    Every session: 

    source scripts/runpod_init.sh

    First-time setup: 

    scripts\setup_env.bat

    Every session: 

    conda activate gnss-sim

    Running a Simulation¶

    A simulation consists of two processes: the server (GPU signal generator) and the client (tester/recorder).

    Step 1 — Start the Server¶

    python -m gnss_sim.framework.server --params config/simulator_params.yaml

    The server binds to port 5555 (ZeroMQ) and waits for client commands. It also exposes Prometheus metrics on port 8000.

    Server startup output 
    GNSS Simulator Server started on port 5555
Prometheus metrics exposed on port 8000

    Step 2 — Run the Client¶

    In a separate terminal (same machine or networked):

    python experiments/rcv_pocketsdr_phase5.py \
    --scenario config/scenario_config.yaml \
    --params config/simulator_params.yaml

    The client will:

    1. Connect to the server on tcp://127.0.0.1:5555
    2. Send INIT with the scenario configuration
    3. Poll STATUS until the server reports READY
    4. Request frames in batches (default: 10 frames per request)
    5. Post-process: add thermal noise, apply AGC, quantize
    6. Write output to .bin files ready for any GNSS receiver
    7. (Optional) Run PocketSDR for acquisition and tracking verification

    Step 3 — Review Results¶

    After the client completes, check the output directory (default: simulation_results/):

    File Description
    ant0.bin Quantized IQ data for antenna element 0
    ant0.bin.tag PocketSDR metadata (sample rate, format, frequencies)
    rcv_ant0.log PocketSDR tracking results

    Configuration Overview¶

    The simulator uses two YAML files:

    File Purpose Who edits it
    scenario_config.yaml What to simulate — time, location, satellites, jammers, antennas User / test engineer
    simulator_params.yaml How to simulate — sample rates, filter parameters, tolerances System designer

    See the Configuration Reference for full details.

    Quick Test: Static GPS L1¶

    Create a minimal scenario config to test your setup:

    config/my_test.yaml
    scenario:
  start_epoch_utc: "2026-01-15T12:00:00Z"
  duration_s: 10.0
  rx_latitude: 32.08
  rx_longitude: 34.78
  rx_altitude_m: 50.0
  elevation_mask_deg: 10.0

  satellites:
    rinex_nav_file: null  # auto-download
    constellations:
      GPS_L1:
        enabled: true
        common_power_dbm: -130.0

  antenna_array:
    elements:
      - id: "ANT_1"
        lever_arm: [0.0, 0.0, 0.0]
        orientation: [0.0, 0.0, 0.0]

    Run the client with:

    python experiments/rcv_pocketsdr_phase5.py --scenario config/my_test.yaml

    Auto RINEX Download

    When rinex_nav_file is set to null, the server automatically downloads the appropriate RINEX navigation file from online archives based on the start_epoch_utc date.
